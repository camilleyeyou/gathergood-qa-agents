Create `tests/api/test_guest_email.py` with 6 tests covering guest list and email settings.

**Imports:** pytest, unique_email from factories.common, event_title/tier_name from factories.common, assert_status from helpers.api, Settings from settings

**Module-scoped fixture — `email_test_event`:**
TEMAL-03 (bulk email) requires an event with at least one attendee. Create a module-scoped fixture that:
1. Creates a new event via `auth_client.post(f"/organizations/{org['slug']}/events/", json={...})`
2. Registers it in `teardown_registry["event_ids"]`
3. Creates a free tier on the event
4. Registers tier in `teardown_registry["ticket_tier_ids"]`
5. Publishes the event via POST `.../publish/`
6. Completes a free checkout via POST `/checkout/` with `action="complete"`, `billing_name="Email Test"`, `billing_email=unique_email()`, `billing_phone=""`
7. Returns dict with `event` and `free_tier` keys

This avoids depending on Phase 3's `completed_order` fixture chain.

**Tests:**

`test_guest_list_view` (TGUST-01):
- GET `/organizations/{org['slug']}/events/{email_test_event['event']['slug']}/guests/`
- Assert status 200
- Assert response is a list: `isinstance(data, list)`
- Assert `len(data) >= 1` (at least the attendee from the fixture checkout)

`test_guest_list_csv` (TGUST-02):
- GET `/organizations/{org['slug']}/events/{email_test_event['event']['slug']}/guests/csv/`
- Assert status 200
- Assert `"Name"` in response text
- Assert `"Email"` in response text
- Assert `"Ticket Tier"` in response text
- Assert `"Confirmation Code"` in response text
- Assert `"Checked In"` in response text

`test_view_email_config` (TEMAL-01):
- GET `/organizations/{org['slug']}/events/{event['slug']}/` (use the session `event` fixture — DRAFT event is fine for reading config)
- Assert status 200
- Assert `"email_config"` in response data
- Note: email_config may be `{}` by default — that's fine, we just verify the field exists

`test_update_email_config` (TEMAL-02):
- PATCH `/organizations/{org['slug']}/events/{event['slug']}/` with `{"email_config": {"send_confirmation": True, "send_reminder": False, "send_notification": True}}`
- Assert status 200
- Assert `updated["email_config"]["send_confirmation"] is True`
- Assert `updated["email_config"]["send_reminder"] is False`
- Assert `updated["email_config"]["send_notification"] is True`

`test_bulk_email_send` (TEMAL-03):
- POST `/organizations/{org['slug']}/events/{email_test_event['event']['slug']}/emails/bulk/` with `{"subject": "Test Bulk Email", "body": "This is an automated test email."}`
- Assert status is 200 or 201 (accept either — the exact success code was not verified in research)
- If 400 and "No attendees" in response text, fail with message about fixture not having attendees

`test_email_log` (TEMAL-04):
- GET `/organizations/{org['slug']}/events/{email_test_event['event']['slug']}/emails/log/`
- Assert status 200
- Assert response is a list: `isinstance(data, list)`
- Note: list may be empty or have the bulk email from TEMAL-03 — assert type only, not content

**Key patterns:**
- Each test has `@pytest.mark.req("TXXX-XX")` marker
- TEMAL-01/02 use the session-scoped `event` fixture (DRAFT — no orders needed for config)
- TGUST-01/02, TEMAL-03/04 use `email_test_event` (needs attendees)
- No `time.sleep()` — API calls are synchronous
